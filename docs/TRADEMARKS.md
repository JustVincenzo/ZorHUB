# Trademark and Official Project Notice

This document defines how the name, identity, and official status of **ZorHUB** may be used.

It does not replace the project license. The project license defines the legal rights to use, copy, modify, and distribute the code. This document defines what may be presented as official ZorHUB.



## Official project

**ZorHUB** is the name of the official project maintained by its original author and maintainer.

The official repository is:

```text
https://github.com/Vynzaro/ZorHUB
```

Only releases, documentation, packages, extensions, and source code published through the official repository or maintainer-approved channels are considered official ZorHUB material.



## Project name

The name **ZorHUB** identifies the official project.

Forks, modified versions, third-party builds, redistributed packages, and community extensions may refer to ZorHUB for compatibility or attribution, but they must not present themselves as the original or official ZorHUB project unless explicitly approved by the maintainer.



## Official use

The ZorHUB name may be used officially for:

- The official ZorHUB repository.
- Official ZorHUB releases.
- Official ZorHUB documentation.
- Official ZorHUB packages.
- Official ZorHUB extensions.
- Maintainer-approved distribution channels.

Only the maintainer may decide what is considered official.



## Forks and modified versions

Forks and modified versions are allowed under the terms of the project license.

However, forks and modified versions must clearly state that they are unofficial unless they have been approved by the maintainer.

Recommended wording:

```text
This is an unofficial modified version of ZorHUB.
It is not maintained, reviewed, or approved by the original ZorHUB maintainer.
```

Modified versions should not be distributed in a way that confuses users into believing they are using the official ZorHUB release.



## Naming for unofficial versions

Unofficial modified versions should use a clearly distinguishable name or label.

Allowed examples:

```text
ZorHUB Community Build
ZorHUB Experimental Build
ZorHUB Fork by <author>
Unofficial ZorHUB Build
Modified ZorHUB Version
```

These names must clearly indicate that the version is not the official release.



## Prohibited representation

Unofficial versions must not:

- Claim to be the original ZorHUB.
- Claim to be an official ZorHUB release.
- Claim to be approved by the maintainer without permission.
- Hide that the version has been modified.
- Use ZorHUB branding in a way that confuses users.
- Distribute modified builds as if they were official.
- Use the official repository link to imply approval of a fork.
- Use the official logo, screenshots, or release assets in a misleading way.

The goal is not to prevent forks. The goal is to prevent confusion.



## Use of the ZorHUB name in descriptions

The ZorHUB name may be used to describe compatibility, origin, or relationship with the project.

Allowed examples:

```text
Fork of ZorHUB
Based on ZorHUB
Compatible with ZorHUB
Extension for ZorHUB
Unofficial build of ZorHUB
```

Not allowed unless explicitly approved:

```text
Official ZorHUB
Original ZorHUB
ZorHUB Stable Release
Approved ZorHUB Build
Maintainer-approved ZorHUB
```



## Extensions

ZorHUB may support extensions in the future.

Extensions may use the ZorHUB name only to describe compatibility.

Allowed examples:

```text
Extension for ZorHUB
Compatible with ZorHUB
ZorHUB integration extension
```

Extensions must not claim to be official unless they have been reviewed and approved by the maintainer.

Recommended wording for community extensions:

```text
This is a community extension for ZorHUB.
It is not an official ZorHUB extension unless approved by the maintainer.
```



## Official extensions

An extension is official only if it is reviewed, approved, and listed by the maintainer.

Official extensions may use wording such as:

```text
Official ZorHUB Extension
Approved ZorHUB Extension
```

only after explicit maintainer approval.



## Logos and visual assets

Official ZorHUB logos, icons, screenshots, banners, and visual assets should not be used in a way that implies official approval of an unofficial version.

Unofficial projects may use their own modified visual identity, but should avoid confusing users into believing they are using the official ZorHUB project.

If an unofficial version uses ZorHUB visual assets for attribution or compatibility, it should clearly state that it is unofficial.



## Package distribution

Third-party packages, redistributed builds, or modified versions must clearly identify themselves as unofficial unless approved by the maintainer.

Examples of third-party distribution channels include:

- Unofficial Flatpak builds.
- Unofficial `.deb` packages.
- Community repositories.
- Third-party package mirrors.
- Forked release pages.

Recommended wording:

```text
This package is an unofficial distribution of ZorHUB.
For the official version, visit:
https://github.com/Vynzaro/ZorHUB
```



## No false endorsement

Use of the ZorHUB name must not imply endorsement, approval, partnership, or maintenance by the original maintainer unless such approval has been explicitly granted.

This applies to:

- Forks.
- Extensions.
- Packages.
- Websites.
- Documentation.
- Videos.
- Tutorials.
- Modified builds.
- Distribution repositories.



## Compatibility with the project license

This document does not remove or restrict the rights granted by the project license.

Users may still use, copy, modify, fork, and distribute the code under the terms of the license.

However, those rights do not automatically grant permission to present a modified version as the official or original ZorHUB project.

In simple terms:

```text
The license allows code freedom.
This notice protects official identity and prevents confusion.
```



## Official approval

A fork, extension, package, or modified version may be considered official only if the maintainer explicitly approves it.

Approval may be granted for:

- Official extensions.
- Trusted community extensions.
- Approved packages.
- Maintainer-reviewed builds.
- Official distribution channels.

The maintainer may revoke official status if a project becomes unsafe, misleading, abandoned, incompatible, or inconsistent with ZorHUB’s goals.
